**blueprint/08-ui-design-tokens.md** (BLUE-08):

Per user decision: "Deep jewel tone color palette: emerald, sapphire, ruby, gold on dark backgrounds -- premium feel with Islamic geometric art influence."

Sections:

**Color Palette:**
Define exact hex values for:
- Primary: Emerald scale (50-900, 6+ shades) -- main action color
- Secondary: Sapphire scale (50-900, 6+ shades) -- secondary actions, links
- Accent: Ruby scale (3-4 shades) -- alerts, streaks, celebrations
- Gold scale (3-4 shades) -- XP, rewards, legendary items
- Neutrals: Dark backgrounds (near-black with slight warm or cool tint), light text shades
- Semantic: success (emerald-500), warning (gold-500), error (ruby-500), info (sapphire-500)
- Rarity colors: Common (emerald), Rare (sapphire), Legendary (gold with glow effect note)

Use format: `token-name: #HEXVAL` with visual description.

**Typography:**
- Font stack: recommend a pixel-friendly body font and a display font for HUD elements (within Claude's discretion per CONTEXT.md)
- Type scale: define sizes for: heading-xl, heading-lg, heading-md, body-lg, body-md, body-sm, caption, hud-level, hud-xp
- Line heights and letter spacing for each size
- Note: Arabic text considerations for future i18n (RTL-safe font choices)

**Spacing System:**
- Base unit (e.g., 4px or 8px grid)
- Named spacing tokens: xs (4), sm (8), md (16), lg (24), xl (32), xxl (48)
- Component-specific spacing: card padding, list item height, button padding, HUD element margins

**Border Radius:**
- Token scale: none (0), sm (4), md (8), lg (12), full (9999)
- Component mappings: cards (md), buttons (sm), avatar (full), HUD elements (none -- pixel art is sharp)

**HUD-Specific Tokens:**
- Pixel art rendering: note that Skia FilterQuality.None is required for crisp pixels
- HUD overlay UI: semi-transparent dark backgrounds for text overlays on pixel art
- XP bar colors and gradient definitions
- Level badge styling

**Component Tokens:**
- Cards: background, border, shadow, padding
- Buttons: primary, secondary, ghost, disabled states with colors
- Inputs: border, focus ring, placeholder color
- Lists: separator color, item height, swipe action colors

**Iconography:**
- Style: pixel art icons for HUD, modern line icons for non-HUD screens
- Size grid: 16x16 (small), 24x24 (standard), 32x32 (large), 48x48 (HUD)
- Icon list: what icons are needed (habits, prayers, quests, settings, profile, etc.)

**Motion Tokens:**
- Duration scale: instant (100ms), fast (200ms), normal (300ms), slow (500ms), celebration (800ms)
- Easing: ease-out for entries, ease-in for exits, spring for celebrations
- Specific animations: XP gain (counter roll-up, 300ms), level up (full-screen burst, 800ms), habit complete (checkmark + pulse, 200ms), streak increment (number flip, 200ms)

**Accessibility:**
- Minimum contrast ratios (4.5:1 for text, 3:1 for large text)
- Reduced motion mode: skip animations, instant transitions
- Color-blind considerations: don't rely on color alone for status

**blueprint/09-sound-haptics.md** (BLUE-09):

Per Claude's discretion (from CONTEXT.md). Sections:

**Sound Identity:**
- Overall audio aesthetic: 16-bit chiptune influences, but refined and minimal -- not a constant soundtrack
- Sound is EVENT-DRIVEN, not ambient. Brief, satisfying audio cues tied to specific actions.
- Mood: calm, rewarding, never urgent or anxiety-inducing

**Event-to-Sound Map:**
Table with columns: Event | Sound Description | Duration | Volume | Priority
Events to map:
- Habit completion (satisfying click/chime, ~200ms)
- Streak increment (ascending tone, ~300ms)
- XP gain (coin-like collect sound, ~150ms)
- Level up (triumphant chiptune fanfare, ~1500ms)
- Title unlock (rare/legendary distinction -- more elaborate for higher rarity, 500-2000ms)
- Quest complete (achievement jingle, ~500ms)
- Mercy Mode activation (gentle, warm tone -- NOT alarm/warning, ~400ms)
- Button tap (subtle click, ~50ms)
- Navigation transition (soft whoosh, ~100ms)
- Muhasabah prompt (gentle bell/chime, ~300ms)
- Error/invalid action (soft low tone, ~200ms)

**Haptic Rules:**
Table: Event | Haptic Pattern | Intensity
- Habit complete: single medium tap
- Level up: triple tap pattern (light-medium-heavy)
- Title unlock: long buzz
- Streak milestone: double tap
- Button tap: light tap
- Error: short sharp tap

**Audio Boundaries:**
- Sound respects device silent mode (always)
- Sound can be disabled in settings independent of device settings
- No sound plays during prayer times (if detectable) -- respect for salah
- Focus mode: all sound off except critical notifications

**Volume Mixing:**
- Sound effects: 70% default, user-adjustable
- No background music in v1 (consider for v2)
- Sounds never overlap -- priority queue with newest wins

**blueprint/10-tech-architecture.md** (BLUE-10):

Document the complete technical architecture. Stack is already decided (from CLAUDE.md and PROJECT.md). This section documents HOW it all fits together.

Sections:

**Architecture Overview:**
- High-level diagram (ASCII): Client (React Native) -> Local DB (SQLite) -> Sync Engine -> Supabase (Postgres)
- Offline-first principle: SQLite is source of truth, Supabase is backup/sync
- Module boundary diagram showing major subsystems

**Frontend Architecture:**
- React Native (Expo) with Expo Router for file-based routing
- Development builds via EAS Build (NOT Expo Go -- per CLAUDE.md)
- Tab-based navigation: Home, Habits, Quests, Profile
- Screen rendering split: HUD screens use Skia canvas, other screens use standard React Native components with pixel art accents

**Rendering Pipeline:**
- Skia (react-native-skia) for Home HUD pixel art rendering
  - FilterQuality.None for crisp pixel art (no anti-aliasing)
  - Canvas-based rendering for game world scene
  - Overlay UI elements positioned over Skia canvas
- Reanimated for 60fps animations on non-HUD screens
  - XP counter animations, streak increments, level-up effects
  - Shared element transitions between screens
- When to use Skia vs Reanimated: Skia for pixel art rendering, Reanimated for UI animations

**State Management:**
- Zustand with domain-split stores (per CLAUDE.md):
  - `habitStore`: habits, completions, streaks
  - `gameStore`: XP, level, titles, quests
  - `uiStore`: modals, toasts, animation states
  - `settingsStore`: preferences, prayer calc method, notification settings
- Store persistence: Zustand persist middleware -> SQLite (not AsyncStorage)
- Store relationships: when a habit completion triggers XP gain, habitStore calls gameStore action

**Local Database:**
- SQLite via expo-sqlite or WatermelonDB (decision pending Phase 2 spike -- document both approaches)
- Migrations strategy: numbered migration files, run on app startup
- Tables map to data model (reference BLUE-11)
- WAL mode for concurrent read/write

**Privacy Gate Module:**
- Central module that classifies every data operation as PRIVATE or SYNCABLE
- PRIVATE data (worship logs, Muhasabah, salah completions): NEVER leaves device, no sync, no analytics
- SYNCABLE data (XP total, level, settings, profile, titles): eligible for Supabase sync
- Implementation: middleware layer between stores and sync engine
- Privacy classifications documented per-entity in BLUE-11

**Backend (Supabase):**
- Postgres database with Row-Level Security (RLS)
- Auth: email, Apple, Google sign-in via Supabase Auth
- Edge Functions: push notification delivery, sync conflict resolution
- Real-time subscriptions: NOT used in v1 (polling-based sync is simpler for offline-first)

**Sync Engine:**
- Offline queue: changes accumulate in SQLite while offline
- Sync trigger: on app foreground + connectivity detected
- Conflict resolution: last-write-wins for settings, idempotent completions (same habit+timestamp = no duplicate)
- Sync protocol: batch upload queue -> server applies -> return confirmed IDs -> clear local queue

**Prayer Times:**
- adhan-js library for local calculation (no server dependency)
- User selects calculation method (ISNA, MWL, Egyptian, etc.) in settings
- Coordinates from device location (one-time permission) or manual city selection
- Prayer times cached daily, recalculated at midnight or location change

**Notifications:**
- expo-notifications for local notifications (prayer reminders, Muhasabah prompt)
- Push notifications via Supabase Edge Functions (v1: prayer reminders only if local notification missed)
- Notification scheduling: calculate prayer times -> schedule 5 local notifications for the day

**Analytics / Telemetry:**
- Reference BLUE-12 for event definitions
- Privacy-safe: no worship data in analytics, no PII
- Likely PostHog or similar privacy-respecting analytics (or custom Supabase events table)

**Configuration:**
- Feature flags: simple JSON config in Supabase (or hardcoded for v1)
- Remote config for quest rotation seeds, XP adjustments (post-launch tuning)

**i18n Infrastructure:**
- i18next for future RTL/Arabic support (per CLAUDE.md)
- All user-facing strings externalized from day 1
- Default: English with Arabic terms inline

**Directory Structure:**
Propose the recommended project directory structure:
```
src/
  app/           # Expo Router screens
  components/    # Shared UI components
  features/      # Feature-specific components
  stores/        # Zustand stores
  services/      # API, sync, prayer times
  game/          # Pure TS game engine (no React imports)
  db/            # SQLite schema, migrations
  types/         # Shared TypeScript types
  tokens/        # Design tokens
  i18n/          # Translation files
  utils/         # Helpers
```

Entities to define:

1. **User** (SYNCABLE -- profile data, no worship content)
   - id, display_name, active_title_id, current_level, total_xp, created_at, updated_at

2. **Habit** (SYNCABLE -- habit definitions are not worship data)
   - id, user_id, name, type (preset/custom), frequency, time_window_start, time_window_end, difficulty_tier, base_xp, status (active/paused/archived), sort_order, created_at, updated_at

3. **HabitCompletion** (PRIVATE -- worship data, never leaves device)
   - id, habit_id, completed_at, xp_earned, streak_multiplier_applied, created_at

4. **Streak** (PRIVATE -- tracks worship consistency)
   - id, habit_id, current_count, longest_count, last_completed_at, multiplier, is_mercy_mode, mercy_recovery_target, updated_at

5. **XPLedger** (SYNCABLE -- only total XP syncs, not per-completion source)
   - id, user_id, amount, source_type (habit/quest/muhasabah/mercy_recovery), source_id, earned_at
   - Note: XP ledger entries sync but are decoupled from HabitCompletion. The completion is private, the XP earned is syncable. This is the privacy boundary.

6. **Title** (SYNCABLE -- title definitions are game data)
   - id, name, rarity (common/rare/legendary), unlock_condition_type, unlock_condition_value, flavor_text, sort_order

7. **UserTitle** (SYNCABLE -- which titles a user has earned)
   - id, user_id, title_id, earned_at

8. **Quest** (SYNCABLE -- quest definitions)
   - id, type (daily/weekly/stretch), description, xp_reward, target_type, target_count, expires_at, created_at

9. **UserQuest** (SYNCABLE -- quest progress)
   - id, user_id, quest_id, status (available/in_progress/completed/expired), progress_count, completed_at, created_at

10. **MuhasabahEntry** (PRIVATE -- deeply personal reflection, NEVER synced)
    - id, user_id, prompt_1_response, prompt_2_response, prompt_3_response, tomorrow_intention, xp_earned, created_at

11. **Settings** (SYNCABLE -- preferences)
    - id, user_id, prayer_calc_method, notification_preferences (JSON), dark_mode, sound_enabled, haptic_enabled, updated_at

12. **Niyyah** (PRIVATE -- personal spiritual intention)
    - id, user_id, text, created_at

13. **SyncQueue** (LOCAL ONLY -- sync infrastructure, not user data)
    - id, entity_type, entity_id, operation (create/update/delete), payload (JSON), created_at, synced_at

**API Contracts (Supabase endpoints for Phase 7):**

Define request/response schemas for sync endpoints:
1. `POST /sync/push` -- Batch upload sync queue items
   - Request: `{ items: SyncQueueItem[] }`
   - Response: `{ confirmed_ids: string[], conflicts: ConflictItem[] }`

2. `GET /sync/pull?since={timestamp}` -- Pull changes since last sync
   - Response: `{ entities: { [type]: Entity[] }, server_timestamp: string }`

3. `POST /auth/register` -- Create account
4. `POST /auth/login` -- Sign in
5. `DELETE /user/data` -- Data export/deletion (GDPR-style)

**Versioning and Migration Strategy:**
- Schema version number in a `schema_version` table
- Numbered migration files: `001_initial.sql`, `002_add_quests.sql`, etc.
- Migrations run on app startup, skip if already applied
- Backward-compatible changes preferred; breaking changes require data migration scripts

**Privacy Boundary Rules:**
- PRIVATE entities: exist ONLY in local SQLite, no column in Supabase, no sync queue entry ever created
- SYNCABLE entities: exist in both local SQLite and Supabase Postgres
- The XPLedger is the critical boundary: it records that XP was earned and from what type (habit/quest/etc.) but NOT which specific habit completion generated it. This allows the game economy to sync while worship data stays private.
- Privacy Gate module (Phase 2) enforces these rules at the code level

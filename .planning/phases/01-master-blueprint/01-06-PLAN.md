**blueprint/12-telemetry.md** (BLUE-12):

Per Claude's discretion on event naming conventions (from CONTEXT.md). Must respect privacy constraints.

Sections:

**Privacy Principles:**
- No worship data in telemetry (no salah times, no Quran reading sessions, no Muhasabah content)
- No PII (no names, no emails, no location beyond city-level)
- Only behavioral patterns: did user open app, did user complete A habit (not WHICH habit), did user level up
- User can opt out of all telemetry

**North Star Metric:**
- Daily Active Users who complete 1+ habits (DAU-completion)
- Why: measures both engagement AND the core value prop (building discipline)

**Event Catalog:**
Table: Event Name | Category | Payload | Privacy Safe | Notes
Events to define (15-20 events):
- `app_opened` -- session start, includes day-of-week
- `habit_completed` -- count-only (not which habit), XP earned
- `quest_started` / `quest_completed` -- quest type, XP earned
- `level_up` -- new level number
- `title_unlocked` -- title rarity (not title name, to avoid identifying worship patterns)
- `streak_milestone` -- milestone number (7, 14, 30, etc.)
- `streak_broken` -- streak length at break (not which habit)
- `mercy_mode_activated` / `mercy_mode_completed` -- recovery duration
- `muhasabah_completed` / `muhasabah_skipped` -- no content
- `onboarding_completed` -- time to complete, habits selected count
- `settings_changed` -- which setting category (not value)
- `notification_tapped` -- notification type
- `app_backgrounded` -- session duration

**Retention Metrics:**
- D1, D7, D14, D30 retention
- Weekly active days (how many days per week user opens app)
- Streak length distribution (histogram of active streak lengths)

**Burnout Indicators:**
- Declining daily completions week-over-week
- Muhasabah skip rate increasing
- Session duration shortening
- Mercy Mode activation frequency increasing
- Alert threshold: if 3+ indicators trigger simultaneously, flag for review

**A/B Test Framework:**
- Define 3-5 initial A/B test ideas: quest rotation frequency, Mercy Mode message variants, XP multiplier cap value, onboarding flow length
- Test infrastructure: simple variant assignment via user_id hash

**Anti-Metric Traps:**
- Don't optimize for session length (we WANT short, productive sessions)
- Don't optimize for notification tap rate (leads to spam)
- Don't track individual worship patterns (privacy violation)
- Don't create "engagement score" that becomes an internal iman score

**blueprint/13-qa-balance.md** (BLUE-13):

Sections:

**Test Strategy Overview:**
- Unit tests: pure TypeScript game engine functions (XP calculation, streak logic, quest matching)
- Integration tests: Zustand store interactions, SQLite read/write, Privacy Gate enforcement
- E2E tests: critical user paths (onboarding, habit completion, streak break + recovery)
- Manual tests: visual QA, animation smoothness, pixel art rendering, adab compliance

**XP/Streak Simulation Validation:**
- Reproduce the three player archetype simulations from BLUE-03
- Verify: casual player reaches expected levels at 7/30/90/365 days
- Verify: streak multiplier caps correctly at 3.0x
- Verify: no negative XP scenarios exist
- Verify: Mercy Mode recovery quest XP is balanced (not exploitable)
- Exploit test: can a player game the system by rapidly creating/completing easy habits? Define the mitigation (daily XP soft cap, habit difficulty tiers)

**Timezone Edge Cases:**
Catalog all timezone-related edge cases:
- User completes habit at 11:58 PM, app processes at 12:01 AM -- which day?
- User travels across timezone -- do streaks break?
- Prayer time calculation across DST boundaries
- Day boundary: when does "today" start? (midnight local time, or Fajr?)
- Ramadan fasting across time zones
Define expected behavior for each.

**Streak Edge Cases:**
- User pauses habit mid-streak -- does streak reset or freeze?
- User archives habit with active streak -- what happens?
- User reactivates archived habit -- streak starts fresh or resumes?
- Two habits with same name (custom) -- separate streaks
- Habit frequency change (daily -> weekly) -- streak recalculation?

**Accessibility QA Checklist:**
- Screen reader compatibility (VoiceOver, TalkBack)
- Minimum touch target size (44x44 pt)
- Color contrast ratios (reference BLUE-08 tokens)
- Reduced motion mode (all animations skippable)
- Dynamic type / font scaling support
- RTL layout readiness (for future Arabic)

**Content Sensitivity Checklist:**
The adab compliance review process:
- Every user-facing string checked against the do/don't copy guide
- Specific patterns to flag: "you missed," "you failed," "your iman," "worship score," "prayer streak lost" (shame framing)
- Approved patterns: "your momentum," "your discipline," "recovery available," "continue your journey"
- Religious content (ayat, hadith) must include source references
- All religious content should be reviewed by a knowledgeable person before release (note in doc)
